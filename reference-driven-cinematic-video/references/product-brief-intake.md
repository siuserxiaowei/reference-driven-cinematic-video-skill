# Product Brief Intake

Use this when the user gives a product intro and expects Codex to fill out the rest of the video.

## Minimum Input

If the user provides only a few sentences, extract or infer:

- product name
- product category
- target user
- core problem
- main result or promise
- concrete features
- available assets
- preferred reference style
- desired video length/platform

Do not block on missing details unless the product identity or allowed claims are impossible to infer safely.

## Creative Brief Output

```markdown
# Creative Brief

Product:
Audience:
Primary pain:
Main promise:
Proof points:
Objections:
Tone:
Reference direction:
Assets available:
Assets needed:
Assumptions:
```

## Claim Discipline

- Facts from user material can be used directly.
- Facts from web research need a source link.
- Competitive or market claims need evidence or must be framed as positioning, not fact.
- If a claim is inferred from product behavior, label it as an inference in working notes and avoid overstating it in final copy.

## Product Video Angles

- **Problem-first**: use when the audience feels the pain but does not know the product.
- **Demo-first**: use when the product is visual or workflow-heavy.
- **Proof-first**: use when there are strong numbers, testimonials, or recognizable customers.
- **Category reframing**: use when the product creates a new way of thinking.
- **Reference-led cinematic**: use when the user points to a specific video style and wants that effect.

## Asset Upgrade Rules

- Replace generic icons with real UI, product renders, or generated scenes with clear art direction.
- If screenshots are ugly but useful, crop and stage them instead of showing full noisy screens.
- If no screenshots exist, build clean UI mocks that honestly represent the described workflow.
- If product identity is weak, create a temporary visual system, but label it as draft branding.
